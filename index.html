F.01

Enterprise Renewal-Driver Analysis

Agentic Claude workflow wired to Salesforce · Databricks · Gainsight · Glean → size-controlled logistic regression → director-ready executive report

An end-to-end renewal-driver study run as a single agentic workflow. Claude pulled roughly 1,000 Enterprise renewal events across two years live from Salesforce, Databricks, and Gainsight, synthesized internal context through Glean, ran the statistics, and rendered a brand-matched executive report — a pipeline that would traditionally take an analyst, BizOps, and a designer.

The core is a size-controlled logistic regression: it isolates the few CS activities that stay predictive of renewal after controlling for account size, segment, and tenure, separating real signal from the roughly eight motions commonly assumed to drive retention. I held the line on rigor throughout — reporting findings as predictive after controls, not causal, and reconciling to the certified rate rather than a convenient number.

Reconciled Customer Success's primary renewal metric to the certified Finance/BizOps rate, closing a long-standing reporting gap, and surfaced an eight-figure pool of uncovered renewals. Delivered as a director-ready report and a CS×Renewals operating-model proposal to inform FY27 Enterprise renewal strategy.
// size-controlled logit — real signal vs. assumed drivers
view implementation
renewal-driver-analysis.py
import statsmodels.formula.api as smf

# ~1,000 Enterprise renewals, 2 yrs — pulled live across
# Salesforce + Databricks + Gainsight by the agent
df = load_renewals(systems=["sfdc", "databricks", "gainsight"])

# which activities stay predictive of renewal AFTER
# controlling for account size, segment, tenure?
model = smf.logit(
    "renewed ~ exec_engagement + qbr_cadence + support_health"
    "  + onboarding_score + ...        # ~8 assumed drivers"
    "  + C(segment) + np.log(arr) + tenure_years",  # controls
    data=df,
).fit(disp=False)

# keep only drivers significant after controls — the signal
signal = model.pvalues[model.pvalues < 0.05].index.tolist()
Correlational with controls, reported as "predictive after controlling for…" — never causal. The certified Finance rate is the source of truth, not a convenient number.
F.02

Operations Command Center

React artifact orchestrating Jira · Slack · Gmail · Calendar · Granola via MCP + the in-artifact Anthropic API ("Claude-in-Claude")

A personal operations cockpit with one-tap buttons for every recurring CS Ops job: morning briefing, weekly status, Confluence docs, BRD creation, Jira ticket drafting, Slack thread summaries, QBR data pulls. Each button calls Claude from inside the artifact, attaches the right MCP servers, gathers live data, and returns a drafted output with a preview-before-send flow.

I rebuilt it on useReducer after early versions broke, to fix stale-closure bugs in the async handlers, and added per-step error isolation through a safeMcp() wrapper so one failed integration doesn't sink the whole run.

Became my daily driver and a template teammates adapted. v8 removed the Atlassian MCP connector entirely, since Samsara's GovCloud Jira/Confluence instance is incompatible with Anthropic integrations under Government Cloud restrictions. All queries reroute through the Glean MCP, which a teammate independently confirmed was more accurate.
// in-artifact API call + type-safe MCP handling
command-center.jsx
async function callClaude(system, prompt, mcp = [], tools = []) {
  const body = {
    model: "claude-sonnet-4-20250514",
    max_tokens: 1000,
    system, messages: [{ role: "user", content: prompt }],
  };
  if (mcp.length)   body.mcp_servers = mcp;
  if (tools.length) body.tools = tools;
  return (await fetch(API_URL, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(body),
  })).json();
}

// extract by block TYPE — never by position; MCP
// responses interleave text / tool_use / tool_result
const results = (d) => (d?.content ?? [])
  .filter(b => b.type === "mcp_tool_result")
  .map(b => b.content?.[0]?.text || "");

// per-step isolation: one broken integration ≠ dead run
async function safeMcp(label, fn) {
  try  { return { label, ok: true,  data: await fn() }; }
  catch (e) { return { label, ok: false, error: e.message }; }
}
State runs through useReducer rather than useState to avoid stale closures inside the async MCP handlers, a specific bug I diagnosed and fixed.

F.04

Daily Operations Briefing Agent

Multi-step read-only Claude agent, packaged as the cs-ops-daily-briefing skill · my agent-track capstone

A morning agent that pulls from five systems I'd otherwise check by hand: Calendar, Granola, Gmail, Slack, and Glean (including the Security-approved GovCloud Jira path). It filters, de-dupes, prioritizes, and returns one ~2-minute briefing with a draft reply and source link on every item. I invoke it by typing "run my daily briefing."

It's designed around two failure modes a naive tool would have. Name ambiguity: a note saying "send to Alex" gets resolved against sent DMs and the directory, and it correctly disambiguated four people named Alex in testing. Nagging about done work: it checks sent Slack and email before listing a task, which in one run cut the to-do list from four items to one.

~30–45 min of daily cross-tool checking collapsed to a ~2-minute read, roughly 2+ hrs/week and ~130 hrs/year. It's built for delegation: per-user prompt and config, no shared logins, so a teammate adapts it by swapping one CONFIG block.

Automated Executive Business-Review Scheduling

Gainsight data-design architecture + Rules Engine build · AI-assisted production engineering

The rolling Business Review (BR) automation for the Enterprise CS org, internally the BR CTA Engine v2 / Rolling Assignment Engine. The v2 design computes, per account, the next BR type, days until the next BR, fiscal quarter, workflow status, and eligibility flags. It checks for already-open follow-ups inside the data design itself, which keeps the firing rule a thin, auditable wrapper that schedules renewal and executive review tasks nightly with duplicate prevention built in.

It also fixed a diagnosed production bug: Canada "Plus" tier accounts over $100K ARR weren't getting executive reviews because a tier filter silently excluded them. I corrected it in both the firing and forecasting data designs.

Newly-eligible accounts now get the right task within 24 hours of crossing the 180-day threshold, with no duplicates. The cutover resolved an open ticket (CSOP-12444) as a byproduct and absorbed a mid-flight scope change by exclusion rather than rework.

04 / Principles

Four patterns run through most of the projects above.

01

Build, prove, then standardize or hand off

Most projects followed the same arc: build the internal AI tooling, validate it in production, then standardize it or hand it off as scope grew. The Account Assignment Command Center became an org standard. The Matik admin work and briefing skills were built to pass cleanly to other owners.

02

Engineering under real constraints

I work within the actual limits. I rerouted Jira/Confluence queries off a GovCloud-incompatible connector to the Glean MCP, respected Gainsight's real feature ceilings instead of speccing builds the platform can't support, fixed stale-closure bugs with useReducer, and isolated integration failures with a safeMcp() wrapper.

03

Trust by design

Read-only agents. Grounding rules that allow only facts the system actually fetched, with no invented names or dates. Name disambiguation, checks for work already done, and a human in the loop throughout, where every recommendation is a draft and the person owns the call.

04

Quantified where it counts

$60M+ at-risk ACV recovered. 500+ exec briefings automated. A daily agent worth ~130 hrs/year. A migration scoped from 30 to 19 templates, with the top 7 at ~85% of volume. I measure each build and report it as it is, including the cases where the result is qualitative, like leadership standardizing on a tool.
